Arsitektur Database SIGARDU

SIGARDU adalah aplikasi web client-side yang berjalan di browser pengguna. Tidak ada server pusat sehingga setiap pengguna mengelola datanya sendiri secara lokal. Database utama menggunakan IndexedDB (otomatis fallback ke localStorage) yang dikelola browser. Berikut struktur skema lengkap:

Tipe: Client-side (Browser) Engine: IndexedDB / localStorage Format Penyimpanan: JSON Enkripsi: - Backup: Manual (Export JSON)
Skema Tabel: gardu_measurements
-- Tabel utama hasil parsing dokumen
CREATE TABLE gardu_measurements (
  id           TEXT PRIMARY KEY,
  nogardu      TEXT NOT NULL,
  unitup       TEXT,
  penyulang    TEXT,
  alamat       TEXT,
  daya_kva     REAL,
  fasa         INT,
  daya_pakai   REAL,
  prosen       REAL,
  tdk_seimbang REAL,
  jam_ukur     TEXT,
  tgl_ukur     DATE,
  petugas      TEXT,
  r_total      REAL,
  s_total      REAL,
  t_total      REAL,
  n_total      REAL,
  upload_id    TEXT,
  created_at   DATETIME
);
Skema Tabel: upload_history
-- Metadata setiap file yang diupload
CREATE TABLE upload_history (
  upload_id    TEXT PRIMARY KEY,
  filename     TEXT,
  filesize     INT,
  filetype     TEXT,
  rows_parsed  INT,
  uw_text      TEXT,  -- UIW header
  up3_text     TEXT,  -- UP3 header
  ulp_text     TEXT,  -- ULP header
  periode      TEXT,
  uploaded_at  DATETIME,
  user_agent   TEXT
);
Skema Tabel: activity_log
-- Backlog aktivitas pengguna
CREATE TABLE activity_log (
  log_id       TEXT PRIMARY KEY,
  event_type   TEXT,  -- access | upload | export | wag
  description  TEXT,
  metadata     JSON,
  timestamp    DATETIME
);

-- Index untuk pencarian cepat
CREATE INDEX idx_event_type ON activity_log(event_type);
CREATE INDEX idx_timestamp ON activity_log(timestamp);
Skema Tabel: settings
-- Konfigurasi tag WAG & threshold
CREATE TABLE settings (
  key          TEXT PRIMARY KEY,
  value        TEXT,
  updated_at   DATETIME
);

-- Contoh isi:
-- key='tag_tl_ulp', value='@TL_ULP_MERAUKE'
-- key='tag_asman', value='@ASMAN_MERAUKE'
-- key='tag_tl_har', value='@TL_HAR_MERAUKE'
-- key='th_overload', value='80'

Relasi & Alur Data

1. Upload File: User upload Excel/PDF/PNG → parsing dengan SheetJS/PDF.js/Tesseract.js → data masuk ke upload_history (1 row) dan gardu_measurements (N rows). Setiap row dapat ditrace ke file asal via upload_id.

2. Aktivitas: Setiap event (akses, upload, export, generate WAG) dicatat di activity_log sebagai backlog. Ini yang ditampilkan di menu Backlog Aktivitas.

3. Konfigurasi: Tag WAG dan threshold overload/underload disimpan di settings agar persist antar sesi.

4. Analisis: Semua query analisis (gardu terlambat ukur, overload, asset) dijalankan in-memory di browser dari data gardu_measurements.

Catatan Teknis
  • Storage Limit: IndexedDB ~50MB+, localStorage ~5MB. Cukup untuk ribuan record gardu.
  • Privasi: Data tidak pernah dikirim ke server eksternal. 100% lokal di browser pengguna.
  • Sinkronisasi: Untuk multi-user, dapat di-upgrade ke backend (PostgreSQL + REST API) dengan skema yang sama.
  • Backup: Gunakan tombol "Export Data" (akan ditambahkan) untuk mengunduh seluruh data dalam format JSON.
  • Kompatibilitas: Chrome, Edge, Firefox, Safari modern (≥2020).
